3. NOTES ON FILES
======================================================================

- app.py: Flask backend. Serves UI/API. Manages users/prefs/Caddyfile. Triggers Caddy reload.
  - Uses `_find_matching_brace()` and `_remove_directive_block()` for proper Caddyfile parsing (handles nested braces, strings, comments).
  - Secret key: auto-generated via `secrets.token_hex(32)` if FLASK_SECRET_KEY is not set or uses default insecure key.
  - Session cookies: HTTPOnly=True, SameSite=Lax.
  - `CADDY_CONFIG_FILE` reads from both `CADDY_CONFIG` and `CADDY_CONFIG_FILE` env vars (fallback chain).
  - Stats API: `/api/stats?period=7d&host=` and `/api/stats/hosts`. Backed by `stats_aggregator.py`. No longer reads raw logs on every request — uses SQLite aggregation.
  - `STATS_DB_PATH = APP_DATA_DIR / 'stats.db'` — initialized both at module level (for gunicorn) and in `__main__` (for dev server).
  - Error rate counts only 5xx (not 4xx).
- stats_aggregator.py: Incremental log processor + stats query engine.
  - SQLite DB with `PRAGMA journal_mode=WAL` for safe concurrent access.
  - Two tables: `hourly_stats` (retention 7d) and `daily_stats` (retention 365d). Schema: `(bucket, host, total, status_1xx..5xx, total_duration, total_size, error_count, top_paths JSON, top_uas JSON)`.
  - `process_new_logs(log_file_path)`: Reads new entries from log file using byte offset tracking. On first run, seeks near end (max 50MB/500k lines) to avoid reading huge history. Aggregates into `hourly_stats` with upsert + merge (top_paths/top_uas merged as JSON dicts, top-N truncated).
  - `rollup_old_buckets()`: Throttled (once per 6h). Hourly→Daily for dates >7d old. Daily cleanup for dates >365d old. All in `BEGIN IMMEDIATE` transaction for ACID safety.
  - `get_stats(period, host)`: Queries appropriate table(s) based on period. 24h/7d → hourly. 30d/90d/1y → daily + hourly (recent 7d). Returns same JSON structure as old API but with `period` and `host` fields.
  - `get_available_hosts()`: UNION DISTINCT from both tables.
  - `has_data()`: Quick check if DB has any rows.
  - Log rotation detection: if file size < last offset, resets offset to 0.
- Dockerfile: Multi-stage build. Installs Caddy binary, Python deps, app code. Defines `CADDY_ACCESS_LOG_FILE` env var.
- docker-compose.yml: Deployment orchestration. Defines service, volumes, env vars. Documents `CADDY_ACCESS_LOG_FILE`.
- docker/entrypoint.sh: Init script. No longer depends on `jq` (uses `[ -s ]` instead). Creates default configs, manages permissions.
- docker/supervisord.conf: Manages `caddy` (root) and `flaskapp` (appuser) processes.
  - Caddy runs WITHOUT `--resume` flag (uses Caddyfile on startup).
  - Single `user=root` for caddy (removed duplicate).
  - Flaskapp environment passes both `CADDY_CONFIG` and `CADDY_CONFIG_FILE`, plus `CADDY_ACCESS_LOG_FILE`.
- templates/index.html: XSS protection via `|tojson|replace('<\/', '<\\/')|safe` for initial data script.
- templates/login.html & setup.html: Apply stored theme from localStorage on page load.
- templates/stats.html: Full stats page with period selector (24h/7d/30d/90d/1y), host filter dropdown, Chart.js timeseries (adaptive granularity), summary cards. Fetches from `/api/stats` and `/api/stats/hosts`. Supports URL params `?host=X&period=7d`. Config logging button retained.
- static/script.js: Caddyfile parsing uses `findMatchingBrace()` (proper nested brace handling with string/comment awareness). Validates modal inputs before save. Stats button (📊) opens `/stats?host=X` in new tab.
- static/style.css: Added dark theme styles for auth pages (`.auth-page`).
- caddyfile/Caddyfile: Example Caddyfile for initialization.
- requirements.txt: Flask, Werkzeug, gunicorn.

4. PROJECT STATUS

ACTIVE_BUGS / KNOWN_LIMITATIONS:
- No CSRF protection on POST endpoints (login, setup, API). Would require Flask-WTF or custom token mechanism.
- No login rate-limiting (brute-force possible).
- No password change feature (only admin creation at setup).
- Single-user system (no multi-user support).
- BASE_DIR for file browsing is CWD (`Path('.').resolve()`), which includes app source code. A logged-in user can read users.json, app.py, etc. via `/api/readfile`.
- Read-only Caddyfile path: `caddyfilePath` is always forced to `CADDY_CONFIG_FILE` regardless of user input (by design, but confusing).

ROADMAP:
- Add CSRF protection (see design decision below).
- Add login rate-limiting.
- Add password change feature.
- Restrict `/api/browse` and `/api/readfile` to specific directories only.
- Multi-user/role support.

DESIGN DECISION — CSRF Protection:
- Pattern chosen: Double Submit Cookie (token in JSON body, not in custom header).
- Rationale: CaddyPanel is typically served behind Caddy itself (reverse proxy). A custom header (X-CSRF-Token) could be stripped by some corporate proxies/WAFs. Putting the token in the JSON body avoids this issue entirely.
- Implementation plan:
  1. Backend: Generate `csrf_token` in Flask session, set a non-httponly cookie `csrf_token` via after_request hook.
  2. Backend: Validate `csrf_token` field in JSON body on all POST endpoints (login, setup, API).
  3. Frontend: JS reads cookie via `document.cookie` regex, includes `csrf_token` in every fetch() JSON body.
  4. Templates: Add hidden `csrf_token` field in login/setup forms.
- Why this works with reverse proxies: No custom headers to strip. Caddy passes everything through unchanged. Even if another proxy strips X-* headers, the token in the body is untouched.
- Why not SameSite cookies alone: SameSite=Lax protects against cross-site top-level navigations (GET), but NOT against same-site subrequests or POST from subdomains. Also older browsers ignore SameSite.
- Why not Flask-WTF: Adds dependency, overkill for this project. A small manual implementation is sufficient.

DESIGN DECISION — Stats Architecture (v1.3):
- Pattern chosen: Incremental log aggregation into SQLite (not raw log re-reading).
- Rationale: Raw log files get rotated/truncated, losing history. Re-reading thousands of lines per API call is expensive. SQLite gives ACID, atomic writes, indexed queries, and persistence across restarts.
- New file: `stats_aggregator.py`. No extra Python dependencies (stdlib `sqlite3`).
- Storage: `stats.db` in APP_DATA_DIR (Docker volume).
- Tables: `hourly_stats` (7d retention) + `daily_stats` (365d retention). One row per (bucket, host).
- Processing: `process_new_logs()` uses byte offset tracking. On first run, reads only tail of log (<50MB / 500k lines). Subsequent runs read only new bytes. Detects log rotation (file shrink → reset offset).
- Rollup: `rollup_old_buckets()` runs hourly→daily for dates >7d. Throttled (once per 6h). Full ACID transaction.
- Query: `get_stats(period, host)` selects appropriate table(s). 30d/90d queries combine daily (older) + hourly (recent 7d) with no overlap.
- API changes: `/api/stats?period=7d&host=` replaces old `/api/stats/global`. `/api/stats/hosts` added. Periods: 24h, 7d, 30d, 90d, 1y.
- Frontend: Period selector buttons, host dropdown, adaptive chart (hourly vs daily granularity).
- Per-site stats: 📊 button in configurator opens `/stats?host=example.com`.
